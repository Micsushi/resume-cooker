# RC-009.6: Editor And PDF Preview UI

## Outcome

Keyboard-operable local browser UI loads approved LaTeX, edits with dirty/conflict state, saves
explicitly, builds, and displays current or clearly stale PDF preview.

## Status Gate

**Blocked by RC-009.3 and RC-009.4.**

## Owned Surfaces

- `generator/public/` frontend files or selected local frontend source
- UI component/state tests
- browser automation fixtures
- `ui-design.md`

## Inputs And Outputs

Input: source/build endpoint contracts and public fixture. Output: source editor, status controls,
explicit save/build actions, accessible PDF preview, and recovery messages.

## Scope

Implement open/dirty/save/conflict/build/current/stale/cancel flows. Keep source, controls, and preview
usable by keyboard with labels/live status. Do not add rich-text, rewriting, auto-save, or cloud
features.

## Sequence

1. Build semantic layout and state store from contract fixtures.
2. Add editor load/dirty/save/conflict flows.
3. Add build/cancel/status controls.
4. Add current/stale/missing PDF preview.
5. Add keyboard and screen-reader state tests.

## Acceptance And Verification

- User completes load/edit/save/build/preview without mouse.
- Conflict never overwrites external edit.
- Failed build labels prior PDF stale.
- Component tests and browser automation against public fixture.

## Failure And Privacy Constraints

No raw HTML rendering, hidden autosave, arbitrary path display, or content in browser persistent
storage/logs.

## Handoff Evidence

Flow screenshots with synthetic data, keyboard results, state fixtures, conflict/stale examples, and
manual browser smoke.
