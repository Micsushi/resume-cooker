# RC-004.2: Immutable Fact Comparison

## Outcome

Postflight comparison detects changed identity/contact, education, employer, title, date, and
configured location facts without false failures from formatting-only differences.

## Status Gate

**Blocked by RC-004.1.** Uses deterministic fixtures only.

## Owned Surfaces

- `checker/scripts/compare-lib.mjs`
- `checker/scripts/compare-lib.test.mjs`
- relevant `fixtures/compare/` cases

## Inputs And Outputs

Input: two normalized fact sets. Output: stable blocker or review finding per changed/missing/ambiguous
fact, with field type, entry index, and confidence but no raw value.

## Scope

Require exact normalized equality for contact values. Match education and experience entries using
explicit keys and confidence. Confirmed immutable change is blocker; ambiguous matching is review
warning. Do not infer whether source facts are true.

## Sequence

1. Add one failing test per immutable field family.
2. Implement deterministic entry pairing and confidence.
3. Separate changed, removed, added, and ambiguous findings.
4. Sanitize evidence and recommended fix direction.

## Acceptance And Verification

- Changed email, phone, employer, title, and date fail synthetic cases.
- Formatting-only contact/date changes pass.
- Ambiguous pairs warn instead of silently matching.
- `node --test checker/scripts/compare-lib.test.mjs`
- Run `compare` against valid and fact-change fixtures; expect statuses `pass` and `fail`.

## Failure And Privacy Constraints

Never print fact values, pair entries solely by array position, or turn extractor uncertainty into a
confirmed accusation.

## Handoff Evidence

Covered fields, pairing rules, fixture results, sanitized findings, and known extraction limits.
