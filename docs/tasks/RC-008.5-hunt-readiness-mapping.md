# RC-008.5: Namespaced Flags And C3 Readiness

## Outcome

Hunt maps Resume Cooker and Fletcher evidence into separate flag namespaces and one explicit C3
readiness decision without losing original reports.

## Status Gate

**Blocked by RC-008.3 and RC-008.4.**

## Owned Surfaces

- Hunt readiness/flag/schema/UI/test paths from RC-008.1
- mapping fixtures and operator documentation

## Inputs And Outputs

Input: preflight result, Fletcher result/flags, postflight result, overrides. Output: readiness state,
`resume_cooker.*` flags, `fletcher.*` flags, report references, and override metadata.

## Scope

Define pass/warning/fail/error mapping through C3. Preserve namespaces and schema versions. A
postflight fact regression blocks readiness unless valid override policy explicitly permits it.
Surface incomplete optional evidence distinctly. Do not merge flags by display label.

## Sequence

1. Freeze full mapping table and synthetic examples.
2. Implement namespaced flag conversion.
3. Compute readiness from policy inputs.
4. Expose sanitized operator evidence and recovery.
5. Test override preservation and incompatible schemas.

## Acceptance And Verification

- C3 cannot be ready when required postflight did not complete.
- Fletcher and Resume Cooker flags never overwrite each other.
- Original report and override remain traceable.
- Hunt mapping tests cover every combined terminal state.

## Failure And Privacy Constraints

Do not persist raw evidence excerpts, discard original blocker flags, or let UI display state become
policy source of truth.

## Handoff Evidence

Mapping table, synthetic flags/readiness examples, override trace, operator view, and policy tests.
