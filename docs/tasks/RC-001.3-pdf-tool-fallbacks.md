# RC-001.3: Capability-Aware PDF Tool Fallbacks

## Outcome

PDF extraction and page counting use the same truthful Docker capability definition as PDF builds.

## Status Gate

**Blocked by RC-001.1.** Deterministic unit work can start before RC-001.2.

## Owned Surfaces

- `checker/scripts/text-layer.mjs`
- `checker/scripts/text-layer.test.mjs`
- shared capability module created by RC-001.1

## Inputs And Outputs

Input: PDF path, local Poppler capability, Docker capability, injected runner. Output: extracted text or
page count with tool metadata, or categorized unavailable/failure evidence.

## Scope

Prefer usable local `pdftotext` and `pdfinfo`; use Docker only when its daemon is usable. Distinguish
missing tool, unusable service, execution failure, empty extraction, and malformed page output. Do
not alter text-layer scoring rules.

## Sequence

1. Add fallback tests for absent Poppler, stopped Docker, failed local tool, malformed output, and
   healthy Docker.
2. Replace executable-presence checks with structured probes.
3. Normalize tool metadata and actionable errors.
4. Ensure optional missing capability becomes explicit incomplete evidence, not pass.

## Acceptance And Verification

- Docker fallback never runs against a known unusable daemon.
- Empty extraction and unknown page count remain distinct.
- Unit tests work without PDF utilities.
- `node --test checker/scripts/text-layer.test.mjs`
- `npm run check:local:ats` with no usable PDF tools; expect classified failure or warning per suite.
- `npm run lint`

## Failure And Privacy Constraints

Never embed extracted resume text in error evidence. Do not weaken the strict one-page contract from
D2.

## Handoff Evidence

Fallback matrix, sanitized sample findings, tool metadata shape, and real-runtime paths not exercised.
