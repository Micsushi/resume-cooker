# RC-006.1: Publish Product Decisions

## Outcome

D1 through D8 exist as accepted public records with rationale, consequences, follow-ups, and revisit
triggers.

## Status Gate

**Ready.** Approved decisions are recorded in `docs/product-decisions.md`.

## Owned Surfaces

- `docs/product-decisions.md`
- `docs/tasks/RC-006-contract-decisions.md`

## Inputs And Outputs

Input: approved ATS, page, enforcement, comparison, tester, API, CLI, and UI contracts. Output:
durable decision records referenced by downstream packages.

## Scope

Publish accepted decisions and map implementation consequences. Resolve contradictory choice
language in RC-006. Do not implement downstream behavior or invent credentials/infrastructure.

## Sequence

1. Record status/date/decision/rationale/consequences/follow-ups/revisit trigger for D1-D8.
2. Link every downstream epic and child package to relevant decisions.
3. Remove instructions asking contributors to choose accepted behavior.
4. Run contradiction search.

## Acceptance And Verification

- All eight IDs appear once as accepted records.
- Page, API, tester, CLI, and UI behavior is unambiguous.
- `rg -n '^## D[1-8]:' docs/product-decisions.md`
- `npx prettier --check docs/product-decisions.md docs/tasks/RC-006-contract-decisions.md`

## Failure And Privacy Constraints

Do not store product truth only in private handoff or turn accepted contracts into undocumented code
assumptions.

## Handoff Evidence

Decision links, contradiction-search results, implementation follow-up IDs, and any explicit future
revisit triggers.
