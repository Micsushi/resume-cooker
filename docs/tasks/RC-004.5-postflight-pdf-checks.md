# RC-004.5: Postflight PDF And Text Checks

## Outcome

When a tailored PDF is supplied, comparison actually inspects its existence, page count, text layer,
sections, encoding, protected terms, and parser agreement.

## Status Gate

**Blocked by RC-001.3, RC-002.2, and RC-004.1.** Unit fixtures may use injected PDF-tool results.

## Owned Surfaces

- `checker/scripts/compare.mjs`
- `checker/scripts/compare-lib.mjs`
- comparison tests
- `checker/scripts/text-layer.mjs` only through existing public APIs
- `fixtures/compare/` PDF/text fixtures

## Inputs And Outputs

Input: tailored PDF plus policy and source protections. Output: artifact existence, page, extraction,
section, encoding, protected-term, and parser findings with tool metadata.

## Scope

Replace metadata-only PDF handling with actual inspection. Apply D2 page policy and shared text-layer
checks. A missing optional PDF is visible; a supplied unreadable PDF fails. Do not duplicate text
logic or require PDF tools for source-only unit tests.

## Sequence

1. Add injected valid, over-limit, empty, corrupt, and unavailable-tool cases.
2. Connect `compare` PDF path to shared checks.
3. Add public real-PDF smoke fixture or reproducible generator.
4. Merge parser agreement when available.

## Acceptance And Verification

- Supplied PDF is opened and classified, not merely recorded.
- One-page and unavailable-count behavior matches D2.
- Empty/corrupt text cannot pass.
- `npm test -- --test-name-pattern="postflight|PDF|page"`
- Run compare with valid and over-limit/corrupt fixtures.

## Failure And Privacy Constraints

No extracted content enters logs; generated PDFs remain ignored or synthetic and intentional.

## Handoff Evidence

Tool/result matrix, actual PDF smoke result, page policy evidence, parser status, and ignored artifacts.
