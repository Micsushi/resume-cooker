# RC-005.4: Provider Readiness Classification

## Outcome

Each supported provider is classified `ready`, `experimental`, or `not_validated` from current,
sanitized evidence without changing opt-in or advisory policy.

## Status Gate

**Blocked by RC-005.2 and RC-005.3 for full readiness.** May document one validated and one
unvalidated provider honestly.

## Owned Surfaces

- `README.md`
- `docs/evaluation-suites.md`
- API provider documentation/comments
- private project handoff

## Inputs And Outputs

Input: live smoke evidence, model identifiers, parsing/error results, and skipped cases. Output:
provider readiness table, supported defaults, limitations, and repeatable smoke procedure.

## Scope

Classify auth/model/response compatibility and guardrails. Update stale default model only with live
evidence. Keep API advisory and explicit opt-in. Do not claim provider quality or SLA.

## Sequence

1. Compare provider evidence against RC-005 acceptance criteria.
2. Assign classification and rationale per provider.
3. Update safe manual procedure and cleanup.
4. Record untested paths and expiry/revalidation trigger.

## Acceptance And Verification

- Documentation never says API is required or default.
- Default model names have dated live evidence or experimental label.
- No secret/raw content exists in changed files.
- `npm test -- --test-name-pattern="API|provider"`
- `rg -n -i "api key|openrouter|anthropic"` review of changed docs.

## Failure And Privacy Constraints

Do not equate one successful request with production reliability or copy provider payloads into docs.

## Handoff Evidence

Classification table, evidence dates, models, request totals, remaining risks, and revalidation
trigger.
