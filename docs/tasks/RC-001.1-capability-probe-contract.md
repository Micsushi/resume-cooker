# RC-001.1: Structured Capability Probe Contract

## Outcome

One shared, unit-tested capability API reports whether commands are installed and whether required
services can actually perform work.

## Status Gate

**Ready.** No native tool or running Docker daemon is required; command execution is injected.

## Owned Surfaces

- `generator/scripts/build-lib.mjs`
- `generator/scripts/build-lib.test.mjs`
- `generator/scripts/check-tools.mjs`

## Inputs And Outputs

Input: command name plus injected runner and timeout. Output: `{ available, usable, reason, metadata }`
without resume data or machine-local socket paths.

## Scope

Implement `probeCommand()` and `probeDockerDaemon()`. Reject missing CLI, nonzero status, known
connection errors on either stream, empty server data, malformed responses, and timeouts. Preserve
the distinction between executable presence and usable capability. Do not select build engines or
change PDF extraction in this package.

## Sequence

1. Add failing fixtures for absent, healthy, nonzero, zero-exit error, empty, malformed, and timeout
   responses.
2. Implement structured probes with bounded execution.
3. Make `check-tools.mjs` render probe results without changing its optional-tool exit behavior.
4. Normalize reasons and remove raw daemon paths.

## Acceptance And Verification

- Installed-but-stopped Docker reports `available: true`, `usable: false`.
- Healthy Docker reports both fields true only after valid server evidence.
- Unit fixtures require no Docker installation.
- `npm test -- --test-name-pattern="capability|Docker"`
- `npm run check:tools`
- `npm run lint`

## Failure And Privacy Constraints

Probe must not hang, start Docker, expose environment paths, or make missing optional tools fail root
CI.

## Handoff Evidence

Fixture result matrix, human tool table, exact timeout, changed fields, and statement that no resume
content left the machine.
