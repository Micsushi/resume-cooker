# RC-004.3: Protected Strength Retention

## Outcome

Comparison warns when protected source strengths disappear and accepts explicitly acknowledged,
intentional omissions.

## Status Gate

**Blocked by RC-004.1.** Independent from PDF checks.

## Owned Surfaces

- `checker/scripts/compare-lib.mjs`
- `checker/scripts/compare-lib.test.mjs`
- `fixtures/compare/`
- comparison input/schema documentation

## Inputs And Outputs

Input: source/tailored facts and optional protected-strength/allowed-omission IDs. Output: retained,
lost, intentionally omitted, and ambiguous strength findings.

## Scope

Protect technical skills, selected projects/experience, and explicit high-value claims through stable
fact IDs. Missing unprotected content is not automatically a warning. Define caller-provided
intentional omission mechanism. Do not rely on one hard-coded global keyword list.

## Sequence

1. Define stable strength IDs and omission input.
2. Add retained, lost, allowed, unknown-ID, and ambiguous fixtures.
3. Implement deterministic matching and sanitized findings.
4. Document how Fletcher supplies protected/omitted IDs.

## Acceptance And Verification

- Protected loss warns or fails only according to supplied policy.
- Valid omission suppresses noise while remaining visible in metadata.
- Unknown omission IDs are invalid input, not silent approval.
- `node --test checker/scripts/compare-lib.test.mjs`
- Run compare against lost-strength and intentional-omission fixtures.

## Failure And Privacy Constraints

Do not expose claim text, treat every source term as protected, or let broad omission patterns disable
the gate.

## Handoff Evidence

ID/omission contract, fixture matrix, sanitized examples, and integration requirements for Fletcher.
