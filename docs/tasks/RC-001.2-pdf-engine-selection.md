# RC-001.2: Capability-Aware PDF Engine Selection

## Outcome

Automatic and explicit PDF builds choose only usable engines and return normalized failures before
claiming an artifact exists.

## Status Gate

**Blocked by RC-001.1.** Unit work needs no real TeX or Docker runtime.

## Owned Surfaces

- `generator/scripts/build-lib.mjs`
- `generator/scripts/build-lib.test.mjs`
- `generator/scripts/build-pdf.mjs`
- `generator/README.md`

## Inputs And Outputs

Input: requested engine or automatic mode, source path, output directory, injected capability probes.
Output: selected engine or categorized unavailable-capability error.

## Scope

Keep preference `latexmk`, `pdflatex`, then Docker. Automatic mode must consume `usable`, not command
presence. Explicit Docker requests validate once and fail with one actionable message. Verify output
PDF exists and is non-empty before success. Do not install engines or change LaTeX content.

## Sequence

1. Add selection tests for stale native commands and unusable Docker.
2. Inject capability results into `detectEngine()`.
3. Normalize automatic and explicit unavailable-engine errors.
4. Verify artifact existence after runner success.
5. Document automatic versus explicit behavior.

## Acceptance And Verification

- Unusable Docker is never selected automatically.
- A stale native executable is skipped.
- Explicit unavailable engine exits nonzero without `docker run`.
- Existing native-engine preference remains stable.
- `node --test generator/scripts/build-lib.test.mjs`
- `npm run build:pdf -- --engine docker` with daemon stopped; expect one actionable failure.
- `npm run lint`

## Failure And Privacy Constraints

Never report success from exit code alone, expose source content in errors, or delete an earlier valid
PDF after a failed build.

## Handoff Evidence

Selection matrix, explicit-error sample, artifact verification test, and untested real runtimes.
